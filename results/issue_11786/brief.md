- **Slug:** tag-rename-stale-name-in-listview
- **Defect:** After renaming a tag (Tags button → Organize Tags → rename), an open list view (e.g. People View) keeps showing the **old** tag name in the Tags column. Switching views and back does not refresh it; only restarting Gramps does. (Mantis 11786, reported against People View, 5.1.2.)
- **Success criterion:** After a tag is renamed while a list view is showing a row tagged with it, that row's Tags column displays the **new** name without restarting Gramps or re-opening the tree — i.e. the rename is reflected on the next view refresh following the `tag-update` signal.
- **Repo + branch target:** gramps (core) @ `maintenance/gramps61` — branched from `upstream/maintenance/gramps61`; forward-merged to `master` by the maintainer (INTEGRATION §2: core fixes ride the current maintenance branch, not `master`).
- **Scope:** Invalidate the renamed tag's stale cached display value(s) so list views re-render the new tag name when the `tag-update` signal fires — the one logical change is making `tag_updated` (or the path it drives) drop the model's `(tag_handle, "TAG_NAME"/"TAG_COLOR")` cache entry for the renamed tag before re-rendering rows. / **out of scope:** redesigning the treemodel cache; touching tag *color* refresh behaviour beyond what falls out of the same cache-key fix; non-list views (chart/relationship views); the Tags side panel.
- **Repro instruction:** fixture `example.gramps`. On `maintenance/gramps61`: open People View, ensure a person carries a visible tag; Tags → Organize Tags → select that tag → Rename to a new string → OK. Observe the Tags column in People View still shows the old name (bug). Verified root cause by reading: the tag display name is cached keyed by the **tag** handle — `get_cached_value(tag_handle, "TAG_NAME")` at `gramps/gui/views/treemodels/peoplemodel.py:511-520` (mirrored in `citationbasemodel.py:317-324`, `eventmodel.py:190`); on rename the DB emits `tag-update` → the view's `tag_updated` (`gramps/plugins/lib/libpersonview.py:535-550`) resolves the **person** backlinks and calls `row_update` (`gramps/gui/views/listview.py:927`), which clears the cache for the **person** handles only. The `(tag_handle, "TAG_NAME")` entry is never cleared, so re-render reads the stale old name; `clear_cache(handle)` exists at `gramps/gui/views/treemodels/basemodel.py:46`.
