- **Slug:** viewmanager-empty-views-indexerror
- **Defect:** Hiding **all** view plugins ("Loaded Plugins") in the Plugin Manager makes Gramps crash at startup with `IndexError: list assignment index out of range`. With no view plugins available, `ViewManager.get_available_views()` returns `[]`, so `views_to_show([])` returns `(0, 0, [])` (`gramps/gui/viewmanager.py:1832` — `default_cat_views = [0] * len(views)` collapses to `[]`); `init_interface` sets `self.current_views = []` (`:641`) then calls `goto_page(0, 0)` (`:645`), which executes `self.current_views[cat_num] = view_num` on the empty list (`:927`) and raises. The same empty-view state then cascades into `__change_page` (`:1049-1051`) and `config_view` (`:1501`). Confirmed by reporter (4.2.0beta2) and re-confirmed by callmedave on 5.2.3; lines unchanged on the target branch.
- **Success criterion:** With zero available views (every view plugin hidden), Gramps initialises the main interface without raising any exception — the navigation path returns cleanly and the app opens with no active page instead of crashing. The shipped regression test drives the **production** startup/navigation path with an empty available-view set: pre-fix it raises `IndexError: list assignment index out of range`, post-fix it completes without raising.
- **Invariant to restore:** The view-navigation path must be **total over the number of available views, including zero** — selecting/activating a page must not assume `self.views` / `self.current_views` is non-empty, and an empty view set must resolve to a no-active-page state rather than an unhandled exception. (Internal Gramps robustness invariant — degenerate-input totality; not borrowing PEP/GTK authority. Not a `docs/principles.md` §6 active category: this is a non-structural behavioural bug fix, so §1.1 minimalism governs and no catalogue citation is mandated. Rationale stated per §7.)
- **Repo + branch target:** gramps-project/gramps @ maintenance/gramps61 (core fix; branch from `upstream/maintenance/gramps61`, forward-merged to `master` — INTEGRATION §2).
- **Surfaces:** gui — the defect lives in `gramps.gui.viewmanager` and is observed at GUI startup; verification is a focused unit test of the production view-selection/navigation decision (a full restart-after-hiding-all-plugins E2E is impractical and the core `T3-interface` gate is smoke-only/advisory). A manual GUI repro stands as the human-facing confirmation.
- **Scope:** Remove the startup crash that occurs when the available-view set is empty — the GUI must initialise its interface and open the main window with no active page instead of raising `IndexError` (and instead of the follow-on `__change_page` / `config_view` failures rooted in the same empty-view state). / out of scope: Plugin-Manager UX changes (e.g. warning the user before hiding the last view), forcing "at least one view always loaded", and any change to plugin-hiding semantics or to which views are registered — those are alternative designs raised in the thread, not part of restoring startup totality.
